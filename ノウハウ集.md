# AdoSQL ノウハウ集

本ファイルは、AdoSQL の実装・運用で得た知見をまとめたメモである。仕様書や README の補足として、実装時に迷いやすい点を優先して記載する。

## 1. ビルド・開発環境

- Visual Studio 2026 (v18) では、`vcvars64.bat` / `vcvars32.bat` を先に実行してから CMake を使うと安定する。
- CMake Generator は `Visual Studio 18 2026` を使用する。
- x64 と Win32 は別ビルドディレクトリに分ける (`build` / `build32`) と管理しやすい。
- C ランタイムは `/MT` (Release) / `/MTd` (Debug) で静的リンクにしているため、配布先で VC++ 再配布の依存が減る。

## 2. PowerShell 実行時の注意

- PowerShell はカレントディレクトリの実行ファイルを暗黙実行しないため、`adosql` ではなく `.\adosql.exe` で起動する。
- 例:
  - `.\adosql.exe .\database.accdb`
  - `.\adosql.exe .\database.mdb`
- `cmd.exe` では `adosql.exe ...` でも起動しやすいが、README では PowerShell 互換を優先して `.\` 付きで案内する。

## 3. ACE / Jet とビットネス

- 64bit EXE では Jet 4.0 は使えない。基本は ACE 16.0 -> 12.0 の順で接続を試行する。
- 32bit EXE では `.mdb` 向けに Jet 4.0 フォールバックを入れられる。
- 実行ファイルのビット数と、インストール済み ACE のビット数が不一致だと接続失敗しやすい。
- エラー文面は「ACE のインストールとビット数一致」を案内する形にしておくと現場での切り分けが速い。

## 4. 文字コード・文字化け対策

- 内部処理は UTF-16 (`wchar_t`) 基準、コンソール I/O は UTF-8 の方針。
- `adosql.ini` と `@` スクリプトは UTF-8 前提、BOM 付き入力も許容している。
- 表示省略記号に Unicode の `…` を使うと、端末環境により文字化けする場合がある。表示系は ASCII (`...`) を優先した方が安全。
- `\r\n` / `\n` / `\r` は可視化して表示 (`<crlf>`, `<lf>`, `<cr>`) すると、データ内容の把握がしやすい。

## 5. REPL 入力モデルの実装方針

- SQL 文は `;` 確定で実行、空行でバッファ破棄というモデルが扱いやすい。
- システムコマンド (`help`, `tablelist`, `describe`, `set`, `show settings`, `prompt`, `commit`, `rollback`, `cls`, `clear` など) は `;` なし即時実行にして操作性を上げる。
- 入力長は 2048 文字上限。超過時は「破棄してやり直し」を徹底すると挙動が明確になる。

## 6. レイアウト整形のコツ

- `linesize` の値だけでなく、実際のコンソール横幅も取得し、実効幅を `min(linesize, consoleWidth)` にする。
- 列表示は「ヘッダ」「区切り線」「データ行」を同一ロジックで生成すると崩れにくい。
- 数値列は右寄せ、文字列列は左寄せを基本にすると見やすい。
- 列幅は過剰に広げず、上限を設ける (例: 40〜60 文字) と横スクロールや見切れを抑えられる。

## 7. ADO / ADOX 実装上の注意

- `#import` で生成される型名・列挙子は環境差が出ることがある。ビルドが不安定な箇所は明示値や互換的な呼び方を使う。
- `Recordset` の EOF 判定など、`#import` の rename 設定に依存する箇所はコンパイルエラーの温床になりやすい。
- `showsql` や `describe` は ADOX 取得に失敗した場合の OpenSchema フォールバックを持つと、実 DB 差分への耐性が上がる。

## 8. Issue 運用のコツ

- 仕様の章番号を Issue 本文の先頭に書くと、実装と仕様の対応関係を追いやすい。
- 「完了条件」を明確に書くと、close 判断がぶれにくい。
- 完了したら close、未完了はコメントで現状を残す。特に「未実装（将来）」は明示して誤解を防ぐ。
